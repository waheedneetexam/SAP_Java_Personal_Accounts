## Status: ✅ Implemented

## Architecture

### Node.js CAP Layer
| Profile | DB | Command |
|---|---|---|
| `development` (default) | SQLite in-memory | `npm run start:sqlite` |
| `production` | SAP HANA | `npm run start:hana` |

### Java Spring Boot Layer
| Profile | DB | Command |
|---|---|---|
| `default` | H2 in-memory | `mvn spring-boot:run` |
| `hana` | SAP HANA | `mvn spring-boot:run -Dspring-boot.run.profiles=hana` |

---

## Configuration Implemented

### `.cdsrc.json`
- Default: `sqlite` + `@cap-js/sqlite`
- `[development]`: explicit SQLite config
- `[production]`: `hana` + `@cap-js/hana`

### `package.json`
- `@cap-js/sqlite` as devDependency
- `@cap-js/hana` as production dependency
- Scripts: `start:sqlite`, `start:hana`, `build:hana`

### `application.yaml` (Java)
- `default` profile: H2 in-memory, mock auth (alice/password)
- `hana` profile: no SQL init, no mock auth, CF VCAP_SERVICES auto-config

### `application-hana.yaml` (Java, NEW)
- Excludes H2 auto-config
- Disables SQL init (HDI manages schema)
- Disables mock security

### `srv/pom.xml`
- H2 dependency scope: `test` (not `runtime` — keeps H2 out of production JAR)

### `db/package.json`
- Node engine: `>=18.0.0` (was `^24.0.0`, which blocked npm install on Node 18/20/22)

### `db/undeploy.json`
- Added `.hdbrole` and `.hdbgrants` undeploy patterns

## HANA Schema

Actual HANA schema is managed by HDI artifacts generated by:
```bash
npm run build:hana        # generates db/src/gen/
cd db && npm run start    # deploys to HDI container
```

Reference DDL: `srv/src/main/resources/schema-hana.sql`

Key H2 → HANA DDL differences:
- `TIMESTAMP(7)` → `TIMESTAMP`
- No `DROP TABLE IF EXISTS` (HDI manages lifecycle)
- No `IF EXISTS` clauses
